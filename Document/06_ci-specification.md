# 06 CI Specification

## 1. 목적

본 문서는 PBShop 멀티 스택 프로젝트에서 공통으로 유지해야 하는 CI(Continuous Integration) 기준을 정의한다.
언어별 구현 방식은 달라도, 검증 목적과 운영 게이트는 동일한 수준으로 맞추는 것을 원칙으로 한다.

기준 구현체는 다음 두 워크플로다.

- TypeScript/TypeORM: `.github/workflows/typescript-typeorm-ci.yml`
- PHP/Laravel: `.github/workflows/php-laravel-ci.yml`

이 문서는 Java, Python, Go 등 다른 언어 트랙이 추가되더라도 공통 CI 설계 기준으로 재사용한다.

## 2. 기본 원칙

### 2.1 공통 목표

모든 언어 트랙의 CI는 다음 목적을 충족해야 한다.

- 코드 품질 저하를 PR 단계에서 차단한다.
- 핵심 API 회귀를 자동으로 검증한다.
- 데이터베이스 마이그레이션 안정성을 검증한다.
- 성능 및 운영 리스크를 사전에 탐지한다.
- 계약 검증, 문서 산출, 배포 전 게이트를 일관되게 관리한다.

### 2.2 언어별 차이 허용 범위

다음 항목은 언어별로 달라질 수 있다.

- 잡 개수
- 잡 이름
- 사용 러너와 캐시 방식
- 테스트 실행 명령
- DB 기동 방식
- Swagger/OpenAPI 산출 방식

다만 다음 항목은 결과적으로 반드시 커버되어야 한다.

- 품질 검사
- 핵심 회귀 검사
- DB/마이그레이션 검사
- 계약 또는 문서 산출 검사
- 성능 smoke 또는 이에 준하는 성능 검증 자산
- 운영/보안/권한 경계 검증 경로
- 릴리스 전 수동 게이트

## 3. 공통 트리거 규칙

### 3.1 자동 실행

모든 언어 트랙은 최소한 다음 이벤트에서 자동 실행되어야 한다.

- `pull_request`
- `push`

자동 실행 범위는 비용과 소요 시간을 고려해 핵심 검증 세트로 제한할 수 있다.

### 3.2 수동 실행

모든 언어 트랙은 `workflow_dispatch`를 지원해야 한다.

수동 실행은 다음 목적에 사용한다.

- 릴리스 후보 검증
- 운영/보안/회복성 심화 검증
- 대기 시간이 긴 회귀 테스트
- 성능 smoke/extended 실행
- 라이브 스모크 점검

### 3.3 수동 입력 검증

`workflow_dispatch` 입력값은 별도 검증 잡으로 먼저 확인해야 한다.

최소 요구사항은 다음과 같다.

- `target_ref` 또는 이에 준하는 입력값을 검증할 것
- 허용된 옵션 외 값은 실패 처리할 것
- 실제 테스트 잡보다 먼저 수행할 것

## 4. 워크플로 계층 구조

### 4.1 입력 검증 계층

대표 잡:

- `validate-dispatch-inputs`

역할:

- 수동 실행 파라미터 유효성 검증
- 허용되지 않은 브랜치/태그/모드 차단

### 4.2 기본 품질 계층

대표 잡:

- `quality`

역할:

- 포맷 검사
- 린트
- 타입 검사 또는 정적 분석
- 단위 테스트 또는 빠른 통합 테스트

### 4.3 자동 회귀 계층

대표 잡 예시:

- TypeScript: `e2e-critical`, `perf-smoke`
- PHP: `api-regression`, `db-integration`

역할:

- 핵심 사용자 플로우 회귀 검증
- API 응답 계약의 핵심 축 검증
- DB 연결 및 마이그레이션 흐름 검증
- 짧은 시간 내 수행 가능한 smoke 성능 확인

### 4.4 수동 확장 계층

대표 잡 예시:

- `release-gate`
- `contract-e2e-manual`
- `migration-validation-manual`
- `migration-roundtrip-manual`
- `security-regression-manual`
- `admin-boundary-manual`
- `rate-limit-regression-manual`
- `dependency-failure-manual`
- `stability-check-manual`
- `perf-smoke-manual`
- `perf-extended-manual`
- `live-smoke-manual`
- PHP 전용 예시: `route-snapshot-manual`

역할:

- 자동 CI에 넣기엔 무겁거나 느린 검증을 수동 게이트로 분리
- 릴리스 직전 점검을 위한 운영성 테스트 실행
- 특정 장애 시나리오와 한계 조건을 독립적으로 검증

## 5. 공통 검증 축

### 5.1 코드 품질

모든 구현체는 최소한 다음 중 해당하는 항목을 포함해야 한다.

- formatter check
- lint
- type check
- static analysis
- 빠른 단위 테스트

### 5.2 기능 회귀

다음 중 하나 이상으로 핵심 기능 회귀를 검증해야 한다.

- 핵심 E2E
- API 회귀 테스트
- 계약 기반 회귀

핵심 범위는 인증, 사용자 접근, 공개 API, 관리자 API 중 프로젝트 핵심 흐름을 포함해야 한다.

### 5.3 데이터베이스 및 마이그레이션

다음 경로 중 최소 하나는 자동, 나머지는 수동이라도 제공해야 한다.

- 마이그레이션 validate
- 마이그레이션 roundtrip
- DB 통합 테스트

DB를 사용하는 트랙은 스키마 생성, 마이그레이션 적용, 롤백 또는 재적용 안정성을 검증해야 한다.

### 5.4 성능

모든 언어 트랙은 최소한 다음 둘 중 하나를 충족해야 한다.

- CI에서 `perf-smoke` 자동 실행
- 수동 실행 가능한 성능 smoke 자산 제공

가능하면 다음 자산도 유지한다.

- soak
- spike
- 검색/정렬/가격 비교 등 도메인별 성능 시나리오

### 5.5 보안, 권한, 회복성

다음 검증 경로를 수동 또는 자동으로 제공해야 한다.

- 인증/인가 경계 검증
- 관리자 경계 검증
- 레이트 리밋 회귀 검증
- 장애 의존성 시나리오 검증
- 회복성 또는 서킷브레이커 계열 검증

### 5.6 계약 및 문서 산출

다음 중 최소 하나 이상을 지원해야 한다.

- API 계약 검증
- OpenAPI/Swagger 산출
- 라우트 스냅샷

문서 산출물은 아티팩트로 업로드하거나 빌드 결과에서 재사용 가능해야 한다.

### 5.7 릴리스 전 게이트

모든 언어 트랙은 `release-gate` 또는 이에 준하는 수동 게이트를 제공해야 한다.

릴리스 게이트는 최소한 다음 상태를 확인할 수 있어야 한다.

- 품질 단계 통과 여부
- 핵심 회귀 통과 여부
- 마이그레이션 안정성 점검 여부
- 계약/문서 산출 가능 여부
- 라이브 또는 사전 운영 smoke 실행 가능 여부

## 6. 아티팩트 및 결과물 규칙

CI는 실행 결과를 단순 성공/실패로 끝내지 말고, 필요한 산출물을 보존해야 한다.

최소 권장 산출물은 다음과 같다.

- 테스트 리포트
- 커버리지 리포트
- 성능 요약 리포트
- OpenAPI/Swagger 산출물
- 라우트 스냅샷 또는 계약 스냅샷

아티팩트는 재현과 리뷰가 가능하도록 잡 이름과 일관된 명명 규칙을 사용한다.

## 7. 환경 구성 규칙

### 7.1 캐시

언어별 패키지 관리 캐시는 적극 사용한다.

- TypeScript: npm cache
- PHP: composer cache
- Java: maven cache

### 7.2 데이터베이스

프로젝트 특성에 따라 DB 구성은 달라질 수 있다.

- PHP는 SQLite와 MySQL 계층을 분리해 검증할 수 있다.
- TypeScript는 테스트 성격에 따라 인메모리 또는 별도 서비스 조합을 사용할 수 있다.
- Java는 H2/실DB/Flyway 기반 조합을 사용할 수 있다.

중요한 것은 구현 기술이 아니라 마이그레이션과 데이터 계층을 실제로 검증하는 경로가 존재하는지다.

### 7.3 서비스 의존성

성능 또는 통합 검증이 필요한 경우 다음 보조 구성을 둘 수 있다.

- mock server
- 테스트 전용 프로필
- health check 대기 로직
- 샘플 데이터 시드

## 8. 기준 구현체 매핑

### 8.1 TypeScript/TypeORM 기준

TypeScript CI는 다음 축을 대표 구현으로 본다.

- 자동: `quality`, `e2e-critical`, `perf-smoke`
- 수동: `release-gate`, 계약/보안/운영/성능/manual smoke 계열 잡
- 특징: Node 기반 캐시, e2e 중심 회귀, mock server와 성능 smoke 연계

### 8.2 PHP/Laravel 기준

PHP CI는 다음 축을 대표 구현으로 본다.

- 자동: `quality`, `api-regression`, `db-integration`
- 수동: `release-gate`, `api-regression-manual`, `db-integration-manual`, `route-snapshot-manual` 및 확장 수동 잡
- 특징: Composer 캐시, 라우트/DB/API 회귀 분리, 프레임워크 명령 기반 검증

## 9. 공통 완성 기준

새 언어 트랙이 CI를 갖췄다고 판단하려면 다음을 만족해야 한다.

1. PR과 push에서 자동 실행되는 기본 워크플로가 있다.
2. `workflow_dispatch` 기반 수동 게이트가 있다.
3. 입력 검증 잡이 있다.
4. `quality`에 해당하는 기본 품질 계층이 있다.
5. 핵심 회귀를 검증하는 자동 잡이 있다.
6. 마이그레이션 또는 DB 통합 검증 경로가 있다.
7. 성능 smoke 또는 이에 준하는 자산이 있다.
8. 보안/권한/운영 경계 검증 경로가 있다.
9. 계약 또는 문서 산출 경로가 있다.
10. 릴리스 전 수동 게이트가 있다.

## 10. 적용 규칙

이 문서는 잡 개수의 통일을 요구하지 않는다.
대신 검증 목적의 통일을 요구한다.

따라서 Java, Python, Go 등 다른 구현체는 다음 원칙으로 CI를 설계해야 한다.

- 잡 이름은 해당 언어와 프레임워크에 맞게 정한다.
- 실행 명령은 해당 빌드 도구에 맞게 정한다.
- 워크플로 분리는 팀 운영 효율에 맞게 조정한다.
- 단, 본 문서의 공통 검증 축은 빠짐없이 반영한다.
